Purpose

Today's world moves at an unprecedented pace. People in all stages of life find themselves struggling to keep up with the amount of responsibilities they have.

An average mom has to plan their children's soccer team party, get a gift for her boss at work from the team, and plan Thanksgiving dinner for over 30 people. While exhausting, these three events are also costly if she does not make sure she gets paid back. That is where IOU comes in!

A user of this application can:

  • Make groups for various events
  • Add necessary people to all of these groups
  • Keep track of what they've spent and what others have spent
  • View a list of all users and what they owe/are owed
  • Make sure they gets paid back for their efforts!

Walk-Through of the Application

Logging in

This screen allows the user to log-in or out of the application. To log-in, a user will select the "Log In" button and then enter their Google account information. To log-out, a user will select "Log out". However, once logged in, the user may continue to the application by selecting "Continue".

View Groups

This screen allows the user to view all of their groups. A user may navigate back to the log-in, add a group, or click a specific group to view more details.

Add Group

This screen allows a user to add a group. The user must enter the specified information, including other users specified by their gmail account, and then select "Add Group".

View Group

This screen allows a user to view the information of a specific group including name, description, members, and the amount those users have spent. The user may also then select to pay back a user or calculate IOUs.

Add Bill

This screen allows a user to add a bill by specifying the necessary information. The user may submit the bill by clicking "Add Bill."

Calculating IOU

This screen shows the calculated IOUs.

PayBack Bill

This screen allows a user to document that they have paid back another user by specifying the necessary information. The user may submit the information by clicking "Add Pay Back Bill."

User IOUs

This screen allows the user to view all the users they are connected with through out their groups. It will display how much money is owed.

Current View of Architecture

The following class diagram shows the current architecture used in the IOU application. The architecture is a MVP pattern (Model-View-Presenter), which is a variation of the better known MVC architecture.

What is MVP?

  • View is a layer that displays data and reacts to user actions.
    • In this application, these are the XML files.
  • Model is a data access layer.
    • In this application, this is the java files; including logic and data containers. As well as the Realm Database.
  • Presenter is a layer that provides View with data from Model.
    • In this application, this is the Activity classes.

Android uses the MVP pattern because the the view and the model classes do not interact. All changes must go through the activities to change the model/what the user sees.

Design Architecture

Here you can see the design structure of the project that includes class diagrams, activities, and XML files. As shown below, a persistence layer was added to show that the database always exists and is not created during runtime.

The View layer lists the XML files used for visual representation of the various activities.

The Presenter layer outlines the different screens that the user will see, which are determined by activities. Though not described in the diagram below, Activities communicate through intents which creates date flow. This is a built in attribute of Android.

The Model layer describes the core data structures and log. A User represents someone who uses the application. A Bill represents money amounts being paid between users. A Group represents a collection of users and bills. The CurrentUser is a Singleton representation of the user that is signed in to the application.

Design Patterns

The Singleton pattern was used in this project to control which user was logged into the application. Because only one instance of a user can be logged in on a single phone at a time, the Singleton Pattern worked so that different activities could access the current user's information.
